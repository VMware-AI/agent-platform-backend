# Versioned migrations

Reviewed SQL migrations applied to **production** postgres. Production never
auto-migrates — `DB_AUTO_MIGRATE` defaults off in prod (`internal/config`), so
the schema is changed only by applying the files in this directory. Dev/test on
in-memory sqlite still auto-migrate from the ent schema.

Migration files are generated by Ent's native Atlas integration
(`../main.go`), not the `atlas-provider-ent` plugin. Generation needs Go + a
throwaway dev postgres (`ATLAS_DEV_URL`); apply/status need the
[Atlas CLI](https://atlasgo.io). See `atlas.hcl` and the Makefile `migrate-*`
targets.

- Generate after a schema change: `ATLAS_DEV_URL=… make migrate-diff name=add_x`
- Apply to a target DB:           `DATABASE_URL=… make migrate-apply`
- Check drift/status:             `DATABASE_URL=… make migrate-status`

The baseline migration `20260621044227_baseline_plus_litellm_token.sql` captures
the full M1 schema (22 entities + the `role_permissions` / `user_roles` join
tables), including `virtual_keys.litellm_token`. Apply it to a fresh prod DB
before first boot; subsequent schema changes append new files via `migrate-diff`.
